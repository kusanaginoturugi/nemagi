## C-10 ワンショット実行とtmux pane残存出力の干渉

**深刻度: 高**

MVP はワンショット実行を基本とする方針になった（第5節）。
しかしtmux pane は長寿命であるため、ターンをまたいで前回の出力がpane内に残る。

- `capture-pane` で差分収集する際、前ターンの末尾行をどこで切るかが難しい
- プロセス終了後もpane内にシェルプロンプトが出ており、出力終端の判別が曖昧になる

**対策案**

ターン開始時にpane内の現在行番号を記録し、そこを差分収集の基点とする。
または、ターンごとにサブシェルを明示的に起動し、出力をファイルにリダイレクトしてからcaptureする方式も有効。

```
tmux send-keys -t <pane> "codex 'prompt' > /tmp/turn-01/codex.out 2>&1" Enter
```

## C-11 各CLIのワンショット実行可否が未確認

**深刻度: 高**

第5節でワンショット実行を基本方針としているが、各CLIが非対話モードを持つか確認されていない。

- `codex`: `codex "prompt"` でワンショット可能か、`-p` フラグの有無
- `claude`: `claude -p "prompt"` のような非対話モードが存在するか
- `gemini`: `gemini "prompt"` でワンショット可能か

対話モードしか持たないCLIが1つでも存在すると、MVP方針の前提が崩れる。

**対策案**

実装前に3CLIそれぞれの `--help` と公式ドキュメントを確認する。
ワンショット不可の場合は `idle_pattern` + `silence_timeout` にフォールバックする設計を維持しておく。

## C-12 コンテキスト引き継ぎの実装が未定義

**深刻度: 中**

第5節に「文脈維持が必要な場合は、Orchestrator が直近の要約または会話履歴を次ターンのプロンプトへ付与する」と記載されているが、具体的な実装方針がない。

- どこまでの履歴を付与するか（直前1ターン？全履歴？要約のみ？）
- 各CLIのコンテキスト長に応じてどう切り詰めるか
- 履歴付与はデフォルトONかOFFか

**対策案**

MVPでは「前ターンのJudge要約を次ターンの先頭に付与する」方式に限定し、仕様を一文で明記する。複雑な履歴管理はMVP外とする。

## C-13 設定ファイルの形式がTOMLとTSで二重定義

**深刻度: 中**

第19節のディレクトリ構成には `config/agents.ts` と `config/runtime.ts` があり、
第20節の設定ファイル案には `agents.toml` 形式が記載されている。

どちらが正とするか定義されていない。

- **TSファイル**: ビルド時に型チェックが効くが、実行後に変更できない
- **TOMLファイル**: 実行時に差し替え可能だが、パーサが必要

**対策案**

どちらかに統一する。MVPでは TOMLをランタイム設定として採用し、`config/agents.ts` は型定義ファイルに位置づける形が整合的。

```
config/
  schema.ts      # 型定義のみ（TOMLの構造をzodで定義）
  agents.toml    # 実際の設定値
  runtime.toml   # ポーリング間隔等
```

## C-14 Controller UIとOrchestratorのプロセス境界が未定義

**深刻度: 中**

第6節でMain UIをtmux pane外から起動する方針になったが、
Controller UIとSession Orchestratorの関係が明確でない。

- 同一プロセス内で動くのか
- 別プロセスとしてIPCで通信するのか
- UIプロセスが落ちたときOrchestratorはどうなるか

**対策案**

MVPでは同一プロセスとし、`main.ts` でUIとOrchestratorを直接連結する。
別プロセス化はMVP外とし、将来の検討事項として明記する。

## C-15 tmux session名の衝突

**深刻度: 低**

起動時に `tmux new-session` でセッションを作成する設計だが、
同名セッションが既存する場合の処理が未定義。

- 既存セッションを再利用するか
- 強制的に作り直すか
- エラーにするか

**対策案**

セッション名にタイムスタンプまたはセッションIDを付与してユニークにする。
または既存セッションが存在する場合はアタッチして再利用するオプションを提供する。

## C-16 RunHandle型が未定義

**深刻度: 低**

第9節の `AgentAdapter` インターフェースに `run()` が追加されたが、
戻り値の `RunHandle` 型が定義されていない。

**対策案**

`types/` に追加する:

```ts
interface RunHandle {
  pid?: number;
  exitCode: Promise<number | null>;
  timedOut: boolean;
}
```

## C-17 Pane 0 と Pane 4 の役割重複

**深刻度: 低**

第6節のペイン構成で:

- Pane 0: `event log / status viewer`
- Pane 4: `judge ログまたは event log`

両方に "event log" が登場している。

**対策案**

役割を分離して明記する:

- Pane 0: `status viewer`（各agentの状態、ターン進捗）
- Pane 4: `judge output`（評価・比較結果の表示専用）
