# 寝間着システム 懸念点リスト

## C-01 応答完了判定の不安定性

**深刻度: 高**

`無出力タイムアウト + 最大時間` だけでは実用上不安定になりやすい。

- `claude` / `codex` / `gemini` の各CLIはバージョンアップでプロンプト文字列が変わる
- 長い応答中に一時的に出力が止まるとタイムアウト誤検知が起きる
- 複数行コードブロック生成中に無出力期間が発生しやすい

**対策案**

- エージェントごとに `idle_pattern` を設定ファイルで上書きできる構造にする
- タイムアウト値も per-agent で設定可能にする

```toml
[agents.claude]
idle_pattern = "^> $"
max_wait_sec = 120

[agents.codex]
idle_pattern = "\\$"
max_wait_sec = 180
```

## C-02 `capture-pane` のANSIエスケープシーケンス混入

**深刻度: 高**

`tmux capture-pane -p` の出力にはカラーコード等のエスケープシーケンスが含まれる。
Judge LLMへ渡す前にストリッピングしないと評価精度が落ちる。

**対策案**

- `-p` のみで取得（`-e` フラグを使わない）
- 取得後に `stripAnsi` 処理を OutputCollector 内で必ず行う
- これを OutputCollector の責務として設計書に明記する

## C-03 OutputCollector のポーリング間隔未定義

**深刻度: 中**

差分収集の頻度が設計書に記載されていない。

- 速すぎる（100ms以下）: CPU使用率が上がる
- 遅すぎる（3秒以上）: リアルタイム表示がもたつく

**対策案**

- デフォルト 500ms、設定で変更可能にする
- 応答中と待機中でポーリング頻度を切り替える適応型も検討

## C-04 Main UIがtmuxペイン内で動くことによる問題

**深刻度: 中**

Controller UIはtmuxセッションのPane 0で動く設計になっている。
TUIライブラリ（`blessed`/`ink`）はtmux内でも動くが以下の問題がある。

- `SIGWINCH`（端末リサイズシグナル）をtmux経由で受け取れない場合がある
- tmuxのバッファリングによりキー入力が遅延することがある
- ネストしたtmuxセッション内では挙動が異なる場合がある

**対策案**

- 早期に `blessed`/`ink` をtmux内で動作確認する
- 代替として、Main UIは別ターミナルウィンドウで起動しtmuxセッションを外部から制御する構成も検討

## C-05 Judge のタイミング仕様が未定義

**深刻度: 中**

全エージェントの応答完了を待つのか、部分的に完了したらJudgeを起動するのかが決まっていない。

- 1エージェントが極端に遅いと全体のターンが長くなる
- タイムアウトしたエージェントの出力（空や途中）をJudgeに渡すとどうなるか未定義

**対策案**

以下のいずれかを選んで設計書に明記する:

1. **全待機**: 全エージェント完了 or 全員タイムアウトまで待つ
2. **多数決**: N/M が完了した時点でJudgeを起動（例: 2/3）
3. **最大待機時間**: 全体に上限時間を設け、完了分だけでJudgeを実行

## C-06 対話セッション維持 vs ワンショット実行の未決定

**深刻度: 中**

`codex`/`claude`/`gemini` に対して:

- **対話セッション維持**: 文脈が引き継がれる。プロセスの生死管理が必要。CLIの再起動でセッションが切れる
- **ワンショット実行**: 毎回新規起動。制御が単純。文脈はシステム側で管理

この選択でAgentAdapterの実装が大きく変わる。設計書では両方に言及しているが決まっていない。

**対策案**

MVPではワンショット実行を推奨。理由:

- プロセス管理が単純
- 応答完了判定が容易（プロセス終了 = 完了）
- 文脈維持はOrchestrator側でプロンプトに追記する形で対応可能

## C-07 `Agent.capabilities` が未定義

**深刻度: 低**

ドメインモデルに `capabilities` フィールドがあるが内容が定義されていない。
実装時に何を入れるか迷う。

**対策案**

削除するか、以下のような型定義を追加する:

```ts
interface AgentCapabilities {
  multilineInput: boolean;   // 複数行入力対応
  streaming: boolean;        // ストリーム出力
  contextWindow: number;     // コンテキスト長
  supportsImages: boolean;   // 画像入力対応
}
```

## C-08 ログ保存形式がストリーミング再現に不向き

**深刻度: 低**

第13節の保存形式では `claude.txt` のようなフラットテキストで保存する設計になっている。
ストリームのチャンク列を保存しないと後で応答過程を再現できない。

**対策案**

`claude.txt` → `claude.jsonl` に変更し、チャンク単位で保存する:

```jsonl
{"t": 1713000001.234, "chunk": "まず", "seq": 0}
{"t": 1713000001.456, "chunk": "結論から", "seq": 1}
```

フラットテキストも `claude.txt` として別途生成すれば人間が読む用途にも対応できる。

## C-09 tmuxに相当するPythonライブラリがTS/Nodeに存在しない

**深刻度: 低**

Pythonには `libtmux` という tmux 操作ライブラリがある。
TypeScript/Node.jsには相当品がなく、`execa` でのサブプロセス呼び出しになる。

これ自体は問題ないが、`tmux` コマンドのエラーハンドリングや戻り値解析を自前で書く必要がある。

**対策案**

- `tmux` 操作を1ファイル（`tmux-backend.ts`）にまとめて、ここだけに tmux 依存を閉じ込める（設計書通り）
- tmuxコマンドごとのラッパー関数を作り、エラー処理を一元化する
